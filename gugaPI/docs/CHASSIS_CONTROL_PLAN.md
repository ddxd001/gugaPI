# 小车控制开发计划

## 1. 文档目的

本文档用于跟踪电赛小车控制功能的设计、实现、测试和验收状态。
每完成一个实现步骤，应同步更新本文件中的状态、测试结果和遗留问题，确保比赛现场能够快速判断当前固件具备哪些能力。

状态定义：

- `已完成`：代码、编译和对应硬件测试均已完成。
- `进行中`：已经开始设计或实现，但尚未通过全部验收测试。
- `待开始`：尚未进入实现。
- `阻塞`：存在明确外部条件或前置问题，暂时无法继续。

## 2. 总体目标

最终控制系统应支持：

- 稳定控制左右轮转速，并在通信异常时可靠停车。
- 使用 ICM-45686 陀螺仪航向角实现直行航向保持和相对角度转弯闭环。
- 非阻塞执行直行、转弯、等待、停车和循迹等基础动作。
- 使用条件驱动动作序列快速组合比赛流程，不需要为每条路线重新编写状态机。
- 通过 Shell 单独测试动作、修改参数、查看遥测并保存比赛参数。
- 在动作超时、传感器异常、通信错误或人工取消时立即进入安全停车状态。

## 4. 阶段一：完善速度控制

状态：`已完成`

目标：在地面负载下稳定完成低速、高速和变速控制，为航向闭环和动作控制提供可靠基础。

### 4.1 明确速度环行为

状态：`进行中`

当前进度：代码、编译、MotorDriver v2 烧录和主控 I2C 联调已完成；等待架空及地面变速测试。

实现内容：

- 整理速度模式进入、目标变化、方向变化和停车时的状态转换。

- 普通目标转速变化时避免无条件清空全部控制状态。
- 明确积分限幅、输出限幅和低速最小占空比之间的关系。

测试：

- 连续发送相同目标，确认 PID 状态不会被重置。

验收标准：

- 相同目标续租不影响 PWM 输出。

### 4.2 加入目标转速斜坡

状态：`进行中`

- 加速斜率：400～800 RPM/s。
- 减速斜率：600～1200 RPM/s。
- 控制周期内使用整数余量累计，避免低斜率被整数截断。

测试：

- `0 -> 300 RPM` 中速起步。
- `80 -> 300 -> 80 RPM` 连续变速。
- 300 RPM 运行时执行普通停车和急停。

验收标准：

- 300 RPM 起步超调不超过 15%。
- 目标变化期间两轮速度差不持续扩大。
- 急停不被速度斜坡延迟。

## 5. 阶段二：航向闭环

状态：`已完成`

> **重要变更**：原计划使用 GY931（维特智能角度传感器）做航向闭环。实际实现改用 **ICM-45686 陀螺仪 Z 轴 yaw 积分**作为航向源，因为 ICM-45686 通过 SPI 直连、采样率更高（10 ms 周期任务）、延迟更低。GY931 驱动仍保留用于角度调试和 OLED 显示，但**不参与航向闭环控制**。

目标：使用 ICM-45686 陀螺仪 yaw 实现直行航向保持和相对角度转弯，不实现底盘里程计、行驶距离估计或二维位姿。

### 5.1 航向采集与角度处理

状态：`代码完成`

实现内容：

- 10 ms 周期任务（`app_imu`）读取 ICM-45686 加速度/陀螺仪，积分 Z 轴角速度得到相对 Yaw（毫度，`-180000~180000 mdeg`）。
- 减去 `ConfigStore` 里的 `imu_gyro_bias_z_mdps` 偏置。
- 提供最短角度差 `ShortestAngleDiff()`，正确处理 `-180°/180°` 回绕。
- 50 ms 周期 `Heading_Update()` 消费 IMU 数据，内建调度看门狗（任务间隔 > 200 ms 判定数据过期→停车+故障）。
- 记录 `last_run_ms`、`last_status` 和 IMU `valid` 标志。

### 5.2 直行航向保持

状态：`代码完成，待硬件验收`

实现内容（`heading.cpp` `HEADING_HOLD` 模式）：

- 动作启动时锁定当前 yaw 为目标航向。
- 根据最短航向误差生成左右轮差速修正：`left = base - correction`，`right = base + correction`。
- `correction_rpm = error_mdeg * heading_kp / 1e6`（64 位运算防溢出），限幅到 `heading_max_correction_rpm`。
- 支持前进和倒车（base_rpm 可正可负），`kYawSign` 控制修正方向。
- 安全：基础速度过低不特别处理（由 MotorDriver 限幅兜底）；IMU 无效或数据过期 → `FAULT_SENSOR_LOST` 停车；航向误差 > 90° → 判定异常停车。

### 5.3 相对角度转弯控制

状态：`代码完成，待硬件验收`

实现内容（`heading.cpp` `HEADING_TURN` 模式）：

- 支持相对左转和右转，目标角度覆盖 `±180°`，最短路径转向，正确处理回绕（`WrapToSigned180`）。
- 使用剩余角度生成差速转弯速度：`speed = abs_err * heading_kp / 1e6`，限幅到 `[heading_turn_min_rpm, heading_turn_max_rpm]`。
- 接近目标时进入容差区间 `heading_tolerance_mdeg`，保持 `heading_settle_ms` 后判定完成（`at_target` + `at_target_since_ms`），避免单次采样越界即结束。
- 转弯超时 8 s（`kTurnTimeoutMs`）→ `FAULT_DRIVER_TIMEOUT` 停车。

### 5.4 参数与遥测

状态：`代码完成`

实现内容：

- 航向控制参数进入 ConfigStore（v3）：`heading_kp`、`heading_max_correction_rpm`、`heading_turn_max_rpm`、`heading_turn_min_rpm`、`heading_tolerance_mdeg`、`heading_settle_ms`。
- Shell `heading status` 输出 mode、target、error、correction、at_target、last_status。
- Shell `param get/set` 可在线修改并 `param save` 持久化到 FRAM。

## 6. 阶段三：非阻塞基础动作

状态：`代码完成`

目标：所有动作都通过周期更新执行，运行过程中 Shell、传感器、看门狗和急停仍可工作。

> 已通过 `heading.cpp`（HOLD/TURN）、`linefollow.cpp`（FOLLOW）和 `action.cpp`（统一解释器）实现。所有运动原语均以 50 ms 周期任务更新，不阻塞主循环。动作失败/超时/取消均调用 `StopAll()`（Heading_Stop + LF_Stop + Chassis_Stop）。

### 6.1 统一动作接口

动作生命周期：

```text
Idle -> Running -> Success
                -> Timeout
                -> Fault
                -> Cancelled
```

统一能力：

- 启动动作。
- 周期更新动作。
- 查询状态和进度。
- 取消动作。
- 超时处理。
- 安全停车。

验收标准：

- 任意动作均不能阻塞主循环。
- 动作失败、超时和取消都必须停止底盘。
- 同一动作对象完成后可以重新启动。

### 6.2 定时速度动作

实现内容：

- 指定左右轮 RPM 和持续时间。
- 支持直行、倒车、原地转向和弧线运动。
- 作为最简单的 ActionRunner 联调动作。

测试：

- 正常完成。
- 执行中取消。
- 超时和通信故障。

### 6.3 定时航向直行动作

实现内容：

- 输入基础 RPM、持续时间、航向控制参数和超时时间。
- 动作启动时锁定当前 IMU yaw。
- 运行期间使用航向误差修正左右轮目标。
- 到达持续时间后停车，不承诺精确行驶距离。

验收标准：

- 动作持续时间误差满足调度周期要求。
- 结束航向误差不超过阶段二的直行验收标准。
- 连续10次动作无明显航向发散趋势。

### 6.4 按角度转弯动作

实现内容：

- 支持原地左转、原地右转和指定半径转弯。
- 根据 IMU 目标航向和剩余角度减速。
- 使用角度容差和稳定时间判定动作完成。

验收标准：

- 90°转向误差不超过 3°～5°。
- 180°转向误差不超过 5°。
- 左转和右转误差接近，不依赖单向补偿。

### 6.5 等待和停车动作

实现内容：

- 非阻塞等待指定时间。
- 普通停车等待速度降到阈值。
- 急停立即清除动作和运动租约。

## 7. 阶段四：动作序列 ActionRunner

状态：`代码完成（已超出原计划）`

目标：通过动作表快速组合比赛流程。

> **实现已超出原计划**：原计划为线性顺序执行器。实际实现为**条件驱动的指令表解释器**，支持条件跳转和分支，可表达循迹直到丢线、转弯直到到位、按键分支、循环等复杂控制流。

### 7.1 指令表解释器

实现内容（`action.h` / `action.cpp`）：

- 指令表最多 16 条指令，每条指令包含 `op`、`param1`、`param2`、`until`（完成条件）、`on_success`、`on_timeout`（跳转目标）。
- 操作码 `ACT_OP`：`DRIVE`（航向保持直行）、`TURN`（相对角度转弯）、`FOLLOW`（循迹）、`WAIT`（等待）、`STOP`（立即停车）、`BRANCH`（条件跳转，不产生运动）、`END`（序列完成）。
- 完成条件 `ACT_COND`：`TIMEOUT`（param2 ms 后完成）、`HEADING_REACHED`（转弯到位）、`LINE_DETECTED`（灰度检测到线）、`LINE_LOST`（灰度丢线）、`BUTTON`（按键 1 按下）、`IMMEDIATE`（立即为真）。
- 跳转目标：`ACT_NEXT`（on_success=下一条，on_timeout=中止）或索引 0..15（goto）。
- 每条指令完成后调用 `StopAll()` 清除运动状态，再启动下一条，避免 drive→wait 继续行驶等问题。
- 整序列超时 60 s（`kSequenceTimeoutMs`）→ 中止。
- `Fault_HasFault()` → 中止。指令启动失败 → 走 `on_timeout` 路径。

示例流程（Shell 构建）：

```text
run clear
run add drive  80  5000  timeout          next abort   # 直行 5 s
run add wait   0   100   timeout          next abort   # 等 100 ms
run add turn   90  8000  heading_reached  next abort   # 左转 90°
run add drive  80  5000  timeout          next abort   # 直行 5 s
run add follow 80  30000 line_lost        next abort   # 循迹直到丢线
run add stop   0   0     immediate        next abort   # 停车
run add end    0   0     immediate        next abort   # 完成
run start
```

测试：

- 空序列。
- 单动作序列。
- 多动作正常完成。
- 中间步骤超时。
- 执行中取消。
- 完成后再次启动。

### 7.2 Shell 调试接口

实现内容：

- `run add <op> <p1> <p2_ms> <until> <onsuccess> <ontimeout>`：追加指令。
- `run clear`：清空指令表。
- `run start`：启动序列（有故障时拒绝）。
- `run cancel`：中止序列。
- `run status`：查看当前步/总步数/运行状态/当前操作码。
- `run dump`：打印完整指令表。

验收标准：

- 比赛调车时无需重新编译即可单独验证一个动作。
- Shell 命令错误不会启动意外动作。

## 8. 阶段五：循迹动作

状态：`代码完成，待硬件验收`

目标：把灰度传感器处理、循迹控制和赛道事件识别封装成可组合动作。

### 8.1 灰度传感器标定

状态：`代码完成`

实现内容（`linefollow.cpp` `LF_CAL` 模式）：

- `lf cal` 启动 2 s 标定，记录各通道 min/max ADC 值。
- 标定完成后计算每通道 threshold = (min + max) / 2，设 `calibrated = true`。
- `ChannelBlackness()` 归一化各通道到 0..1000（1000 = 完全在线上）。
- 标定参数当前存 RAM（每次上电重新标定），计划后续持久化到 FRAM。

### 8.2 基础循迹控制

状态：`代码完成，待硬件验收`

控制形式：

```text
left_rpm  = base_rpm - correction
right_rpm = base_rpm + correction
```

实现内容（`linefollow.cpp` `LF_FOLLOW` 模式）：

- 加权位置计算：`pos_milli = (sum(i * blackness_i) / sum(blackness_i)) * 1000`，`error_mpos = (pos_milli - 3500) * kLineSign`，范围 `-3500..+3500`。
- `correction = error_mpos * kp / 1e6`，限幅到 `max_correction_rpm`（默认 kp=10000，即 1 通道偏差约 10 RPM）。
- 丢线判定：`total_blackness < kMinLineWeight(1000)` → 保留上次方向修正 `lost_timeout_ms`（默认 500 ms），超时则停车。
- 持续时间到达 `follow_duration_ms` 后自动停车。
- 安全：灰度数据无效 → `FAULT_SENSOR_LOST` 停车；故障 → 停车。

Shell 在线调参：

```text
lf kp <val>           # 设置 kp（0..1000000）
lf maxcorr <val>      # 设置最大修正 RPM（0..500）
lf losttimeout <ms>   # 设置丢线超时（100..10000 ms）
```

### 8.3 赛道事件识别

## 9. 阶段六：比赛调车与参数管理

### 9.1 参数体系

状态：`代码完成`

实现内容（`config_store.cpp`，FRAM v3 布局）：

- 速度环、航向闭环、IMU 偏置和底盘几何参数统一进入 ConfigStore，持久化到 FRAM（地址 0x0000，magic "CFPG"，CRC32 校验）。
- 当前版本 v3，payload 90 字节；兼容加载 v1（66 字节）和 v2（68 字节）历史布局。
- `param set` 修改后显示 dirty 状态，`param save` 显式持久化，`param load` 从 FRAM 重新加载，`param reset` 恢复源码默认值。
- 参数列表（32 项）：
  - 底盘：`left/right_counts_per_rev`、`wheel_radius_mm`、`wheel_track_mm`、`max_wheel_rpm`、`motor_output/encoder_invert_flags`
  - 速度环：`speed_kp/ki/kd`、`speed_max/min_duty`
  - 位置环：`position_kp/ki/kd`、`position_max_rpm`、`position_tolerance_counts`
  - GY931 零点：`gy931_roll/pitch/yaw_zero_mdeg`
  - IMU 偏置：`imu_accel_bias_x/y/z_mg`、`imu_gyro_bias_x/y/z_mdps`
  - 航向闭环：`heading_kp`、`heading_max_correction_rpm`、`heading_turn_max/min_rpm`、`heading_tolerance_mdeg`、`heading_settle_ms`

### 9.2 调车遥测

状态：`代码完成`

实现内容：

- FireWater 协议（VOFA+ 兼容）周期输出 CSV 数据，支持实时画图。
- 通道：`t,mode,step,L_tgt,L_act,R_tgt,R_act,yaw_tgt,yaw,err,corr`
- `telem on [period_ms]` 开启（默认 100ms = 10Hz，范围 50..5000ms）
- `telem off` 关闭，`telem status` 查看状态
- TX 环形缓冲 > 3000 字节时自动丢帧，避免阻塞控制循环

### 9.3 比赛模式

状态：`代码完成`

实现内容：

- 编译时：`feature_competition_config.h` 开启 IMU/LED/BUZZER，关闭 LoRa/诊断/调试日志。
- 运行时状态机：`APP_MODE_COMPETITION_ARMED`（安全静止）→ `APP_MODE_COMPETITION_RUNNING`（序列执行中）→ ARMED。
- 上电后保持安全静止（比赛配置下 chassis 任务禁用，电机 coast）。
- 按键 1 或 `comp start` 启动比赛序列（ActionRunner）。
- `comp stop` 或按键 1 取消序列，返回 ARMED。
- `comp arm` 从开发模式进入比赛模式（测试用）。
- 故障后锁定停车（FAULT），必须 `reset` 复位。
- LED 指示：ARMED 慢闪（1Hz）、RUNNING 常亮、FAULT 快闪（5Hz）+ 蜂鸣器。
